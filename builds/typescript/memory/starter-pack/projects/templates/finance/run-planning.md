# Finance Planning — Procedure

*Procedure for filling out `plan.md` — how your Finance agent turns the goals in `spec.md` into a sequenced, actionable plan and knows when the plan is good enough to start executing. Used every time `plan.md` is built or revised, not a one-time activity.*

## What This Procedure Accomplishes

`plan.md` has two jobs that this procedure delivers on:

1. **Turn defined goals into a sequenced, phased path of action** — the heart of the plan.
2. **Surface the next concrete step and the pull behind it** — so the owner moves rather than plans forever.

Both matter. A great roadmap with no immediate step is paralyzing. A great first step with no path is shortsighted.

## Preservation Rule

When updating `plan.md`, **update sections in place — never replace the whole file**.

**Always keep, even when a section has no owner content yet:**

- Every section header (`## Right Now — Your Next Step`, `## The Roadmap`, `## The Destination`, `## What's Blocking`, `## Changelog`, and any phase or subsection headers already present).
- The italic section description immediately under each header.
- The `*Last updated: …*` line in each section.
- The `**Status:**` line at the top of the plan.
- The `## Changelog` section.

If a section has nothing to add yet, leave its placeholder (`*To be filled...*`) in place. **Do not delete sections, descriptions, or the "Last updated" line.**

**You may remove:** generic example blockquotes (the `> *Example...*` or `> ***Phase...***` example phase blocks) inside a section, once that section has real owner content — unless the examples are still useful for orientation.

**Use today's date** (from your current system context — e.g., the conversation date the owner is having with you right now) for the "Last updated" lines and any changelog entry. Do not invent or carry over a date from an unrelated source.

## When to Run

Run this procedure when:

- **`spec.md` has at least one goal that meets the spec-stage done criteria** — first time planning a goal
- **A new goal lands in `spec.md`** — add phases for it
- **A phase completes** — move forward, fill in more detail on the next phase, update Status and "Right Now"
- **Life changes contradict the plan** — re-sequence, re-size, or revise the destination
- **A blocker clears** — pull the previously-blocked phase into more concrete detail

## Method

How to build phases, sequence them, and keep the plan moving:

- **Source from `spec.md`.** Only build phases for goals defined in `spec.md`. If the spec flagged something as "Worth exploring later," it doesn't get a phase here. The plan serves the spec, not the other way around.
- **Lead each phase with the pull.** Before listing the steps, write what life looks like when this phase is done — in the owner's terms. *"I actually know what's happening"* or *"The foundation is in place"* beats *"Phase 1: Data gathering."*
- **Size the first move for momentum.** "Right Now" should take less than an hour and produce something the owner can see or feel. The point is movement, not coverage. If the first step takes an afternoon or requires three other things first, it's too big.
- **Detail decreases with distance.** Early phases (next 1-3 months) are concrete. Later phases (3+ months out) stay high-level and get filled in as earlier phases complete. Don't pretend to know exactly what month 7 looks like from month 0.
- **Cross-project actions live inline in the phase that needs them.** Call them out as part of that phase's steps (e.g., *"Cross-project: ask Career agent about raise conversation"*). Don't duplicate `spec.md`'s "Where This Connects" — that section is the cross-project thread inventory; the plan is the actual cross-project actions.
- **Use the owner's words for the Destination.** Quote the success criteria from `spec.md`'s "What You Want" — don't rewrite into your own voice. Their words pulled them in; their words will pull them through.
- **Destination changes go to `spec.md` first.** The owner's success criteria live in `spec.md`'s "What You Want." If the destination needs to shift materially, update the spec first (run the interview procedure), then quote the new version in `plan.md`. Don't create a second source of truth.
- **Mark phase status.** Each phase header carries one of: *Not started / In progress / Complete / Blocked / Deferred*. Format: `### Phase 1 — In progress — "pull"`. Update as work moves — when a phase completes, advance the next one to "In progress" and resize/refine "Right Now" accordingly.
- **Name blockers concretely.** Each item in "What's Blocking" names what's blocked AND what would unblock it. Vague blockers don't help — *"need more info"* doesn't tell anyone what to ask. *"Need CPA confirmation of marginal tax rate before Roth vs. traditional decision"* does.

## Done Criteria

Before the plan is ready for the owner to start executing, you should be able to answer YES to all of these:

1. **At least one goal from `spec.md` has phases.** A plan with no concrete phases isn't a plan.
2. **"Right Now" is sized for momentum.** The next step is < 1 hour, this week, and the owner can see it through without help.
3. **The first 1-3 months are concrete.** Phase 1 has real steps, not "figure out a plan."
4. **The destination is in the owner's words.** Quoted from `spec.md`, not rewritten.
5. **Blockers are named.** Anything preventing more detail is in "What's Blocking" with what would unblock it.
6. **Phase status reflects current reality.** Every phase carries an accurate status marker; "In progress" matches what's actually happening right now.

If any are "no" or "kinda" — keep planning. Don't hand off a plan with no first step.

## After Planning

Once a plan meets the done criteria:

1. **Update `plan.md`** — fill in sections with phase content and set "Last updated" dates.
2. **Update Status** at the top of `plan.md` — reflect current state (e.g., "Phase 1 in progress" or "Aligned and ready to execute").
3. **Sync with `spec.md`** — if planning surfaced a gap that belongs in the spec (e.g., owner realized they need to define a new goal or revise an existing one), update `spec.md` too.
4. **Log it in the changelog.** One line at the bottom of `plan.md`: date + what changed + why. New phase added, phase completed, sequencing revised. Skip cosmetic edits.
5. **Hand off to execution.** Point the owner at "Right Now" and let them do it.

## What This Procedure Is Not

- **Not a form.** Don't run the Method bullets as a checklist. They're behavioral defaults applied conversationally.
- **Not exhaustive.** You're not planning every possible phase. You're planning enough that the next 1-3 months are real, and high-level beyond that.
- **Not laminated.** Plans evolve. Run this again every time a phase completes, life changes, or a blocker clears. Re-planning is normal, not exceptional.
