# Project Interview

*Procedure for filling `spec.md` through conversation.*

## Preservation Rule

Update sections in place in `spec.md`; never replace the whole file. Always keep every section header, every italic section description, every `**Status:**` line, every `**Last updated:**` line, and `## Changelog`. Use today's date from system context for Last updated and changelog entries.

## What This Procedure Accomplishes

Build a clear project picture and capture the owner's goals, current reality, constraints, and missing information.

## When to Run

- The project spec is empty or materially stale.
- The owner wants to clarify the project before planning.
- New scope, timing, resource, support, or constraint information could materially change the project direction.

## Method

Start with what the owner brought, then adapt to the project type. For practical projects, map scope, budget, timeline, dependencies, and decision points. For creative or learning projects, map current output, practice rhythm, standards for good enough, and blockers like perfectionism. For sensitive life projects, slow down, ask what progress would mean, and identify support systems.

If the topic clearly belongs in Finance, Fitness, Career, or Relationships, offer to route it there. If the owner wants it here, keep it here.

## Done Criteria

`spec.md` has useful owner-specific content, important unknowns are labeled, and the project direction is not based on unsupported assumptions.

## After Running

Update `spec.md`, summarize material changes, add todos only for concrete next actions, and return to project scope before proposing planning.

## What This Procedure Is Not

It is not therapy, legal advice, medical advice, financial advice, or a substitute for qualified professional support when the project requires it.
