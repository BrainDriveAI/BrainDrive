# Fitness Journal - Agent Context

*Folder scope for owner-entered Fitness follow-up history.*

This folder stores the owner's Fitness journal after `spec.md` and `plan.md` exist. The journal is owner-driven follow-up context, not mandatory daily tracking.

## Preservation Rule

Before writing, correcting, or recovering `journal.md`, read the existing file when it is readable. Never replace the whole file. Append new entries or targeted-edit only the intended entry. If `journal.md` is unreadable or corrupt, create a timestamped recovery file, preserve the original, and tell the owner what happened.

## What Belongs Here

- Owner-provided Fitness progress, blockers, wins, changes, and reflections.
- Fitness context about food, movement, workouts, energy, sleep, recovery, adherence, schedule friction, and plan mismatch.
- Health-adjacent context only as owner-provided Fitness context.
- Concrete commitments in `Follow-up -> Proposed task` for V.1.

## What Does Not Belong Here

- Workflow instructions, prompts, rubrics, reports, source files, or test artifacts.
- Raw copies of Fitness `spec.md`, Fitness `plan.md`, `me/profile.md`, or task lists.
- Invented metrics, diagnoses, treatment advice, medication guidance, or medical certainty.
- Passive tracker, wearable, food database, workout platform, dashboard, or social fitness state.

## Parent And Profile Updates

Fitness `spec.md` owns current goals, current state, constraints, assumptions, and success criteria. Fitness `plan.md` owns ordered next steps, milestones, and plan status. Update those parent files only after the owner agrees to a concise current-truth change. Stable cross-project owner facts go to `me/profile.md` only after confirmation.
